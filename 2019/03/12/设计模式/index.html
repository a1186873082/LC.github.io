设计模式

OO–面向对象编程的设计原则:

  • 单一职责原则(SRP):每个类负责一个功能的实现。
  • 封闭-开放原则(OCP):我们编写程序时应该做到对修改封闭，对扩展开放
  • 依赖倒转原则(DIP):我们应该针对接口编程，而不是针对实现。高层模块不应该依赖于底层模块，二者都应该依赖于抽象。抽象不应该依赖于细节，细节应该依赖于抽象
  • 里氏替换原则(LSP):子类应当可以替换父类并出现在父类能够出现的任何地方
  • 接口分离原则(ISP):一个接口只处理一项相关的事情

常用的设计模式:

  • 单例模式:保证对象在任何时候都只会创建一次，并使用第一次创建的对象。

    例子:使用双重锁技术:将变量设置为volatile就是为了防止重排序问题

    public class Signated {

    private static volatile Signated signated;

private Signated() {}

public static Signated getInstance() {
    if(signated == null) {
        synchronized(Signated.class) {
            signated = new Signated();    
        }
    }
    return signated;
}

    }

    使用类初始化特性保证单例:

    private Signated() {}

  • 静态工厂

    总结:

    1.防止对象被实例化使用时，可以将构造器设置为private类型的。

    2.利用静态工厂方法代替构造器

    利用静态工厂方法可以保证获取对象时不必每次都创建一个新对象(比如单例模式)

    单例模式:

    Class Test{

    Private Test(){

    }

Public static final Test test = new Test();

Public static Test getInstance(){

Return test;

}

}

Class Test{

Private Test(){}

Public Test test;

Public Synchronized static Test getInstance(){

If(test == null){

test = new Test();

}

return test;

}

}

利用静态工厂方法可以避免代码复杂的问题，比如

Public HashMap<String, Object> obj = new HashMap<String, Object>();

可以简化为:

Class MyMap<K, V>{

public static <K,V> MyMap<K,V> getInstance(){

​ return new MyMap<K,V>();

}

}

Class main{

Public static void main(String args[]){

MyMap<String, Object> myMap = MyMap.getInstance();

}

}

此时不用多次的指明类型参数，可以自行推导出来

静态工厂方法的缺点:

1，当不存在public或protected类型的构造器时，无法被其他类继承

2，没有明确标识的实例化对象的方法，要按照规范命名方法，比如getInstance,newInstance,valueOf 等方法名
